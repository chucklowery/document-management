# Architecture Decision: Canonical Semantic Serialization

**Identifier:** ADR-HT-002

**Status:** Accepted

**Date:** 2026-06-27

**Project:** Organizational Knowledge and Work System

## Context

The system must preserve meaning across textual editors, visual editors, APIs, repositories, databases, exports, migrations, publication pipelines, and future replacement implementations.

No editor format, database schema, repository layout, or rendered output can safely serve as the permanent semantic authority for all bounded contexts.

The system requires a canonical serialization for:

- Resource and Resource Revision envelopes;
- Relationships;
- Content Regions and Region Occurrences;
- Immutable Records;
- Trails and Trail Steps;
- Projection Definitions and Projection Instances;
- provenance;
- authority and epistemic status;
- manifests;
- governed extensions;
- independent reconstruction.

The serialization must support stable hashing, deterministic comparison, digital signing, lossless round trips, schema evolution, and human-auditable examples.

## Decision

### 1. The canonical semantic representation is versioned JSON

The canonical interchange and persistence-neutral representation shall be a versioned JSON semantic envelope.

JSON is the authoritative serialized representation for cross-context exchange, manifests, exports, integrity calculation, and reconstruction tests.

This decision does not require every bounded context to use JSON as its internal runtime or database representation.

### 2. Semantic authority belongs to the model, not presentation syntax

Markdown, HTML, PDF, DOCX, visual canvases, dashboards, forms, and editor-specific document trees are authoring, editing, or presentation representations.

They are not the sole semantic authority unless a narrower context contract explicitly says otherwise.

A conforming adapter shall preserve all canonical semantic information required for the object it edits or renders.

### 3. Every envelope declares its contract

A canonical envelope shall contain these top-level fields:

```json
{
  "contract": "okw.resource-revision",
  "contract_version": "1.0.0",
  "identity": {},
  "recorded_at": "2026-06-27T18:30:00Z",
  "payload": {},
  "provenance": {},
  "extensions": {}
}
```

Required top-level semantics:

- `contract` identifies the semantic contract;
- `contract_version` identifies the contract schema version;
- `identity` carries the durable identity envelope from ADR-HT-001;
- `recorded_at` is an RFC 3339 UTC timestamp for serialization or record creation;
- `payload` contains contract-owned semantic data;
- `provenance` contains material lineage and transformation information;
- `extensions` contains governed extension namespaces.

Contracts may require additional top-level fields where their semantics apply universally to the envelope.

### 4. Contract names are stable and implementation-neutral

Contract names use lowercase dotted identifiers under the `okw` namespace.

Examples:

```text
okw.resource
okw.resource-revision
okw.relationship
okw.content-region
okw.region-occurrence
okw.immutable-record
okw.trail-revision
okw.projection-definition
okw.projection-instance
okw.resolution-manifest
okw.publication-manifest
```

Contract names shall not encode:

- programming language;
- service name;
- database table;
- repository path;
- organization-specific deployment details.

### 5. Contract versions use semantic versioning

`contract_version` uses `MAJOR.MINOR.PATCH` form.

- A major version indicates a breaking semantic or structural change.
- A minor version adds backward-compatible fields, enum values, or extension points.
- A patch version clarifies or corrects the contract without changing valid meaning.

Consumers shall reject an unsupported major version unless an explicit migration path is available.

Consumers shall preserve unknown fields from supported major versions when round-tripping an envelope unless policy requires rejection.

### 6. Canonicalization follows JSON Canonicalization Scheme rules

Canonical bytes used for hashing, signing, equality checks, and reproducibility shall follow RFC 8785 JSON Canonicalization Scheme behavior.

Canonicalization includes:

- UTF-8 output;
- no insignificant whitespace;
- deterministic object-property ordering;
- deterministic JSON primitive serialization;
- no duplicate object-property names;
- values constrained to the interoperable JSON data model.

Human-readable files may use indentation and editorial property ordering. Integrity operations shall canonicalize parsed semantic content rather than assume the stored text is already canonical.

### 7. Numbers do not represent unconstrained decimal business values

JSON numbers may be used only when the contract defines their precision and range as safely interoperable.

Money, rates, measurements requiring exact decimal precision, and values outside the safe numeric range shall use explicit string representations with separate unit or currency fields.

Example:

```json
{
  "amount": "125000.00",
  "currency": "USD"
}
```

Timestamps, durations, dates, identifiers, checksums, and semantic versions are strings with contract-defined formats.

### 8. Arrays preserve semantic order

Array order is significant unless the contract explicitly declares the collection unordered.

Unordered semantic collections shall either:

- use an object keyed by stable identity; or
- define a deterministic sort rule before canonicalization.

A serializer shall not reorder an ordered Trail, document block sequence, provenance transformation sequence, or other semantically ordered collection.

### 9. Identity envelopes are explicit

A principal identity reference uses:

```json
{
  "id": "01906c68-0fc1-7b16-bb5a-e861428a1377",
  "kind": "ResourceRevision",
  "owning_context": "KnowledgeAndProvenance"
}
```

A reference may add:

- `resource_id`;
- `revision_id`;
- `relationship_id`;
- `record_id`;
- `selection`;
- `as_of`;
- external mapping information.

Identifiers remain opaque. Their semantic kind and ownership come from explicit fields.

### 10. Typed references replace embedded copies when identity matters

When a semantic object refers to another managed object, it shall use a typed reference rather than copy identity-bearing content as if it were local authority.

A reference shall identify the target and, where applicable:

- exact Revision;
- selection mode;
- historical boundary;
- expected contract;
- relationship role;
- authorization-sensitive resolution behavior.

Cached or embedded display content is derivative and shall identify the referenced source and cache status.

### 11. Provenance is a standard sub-envelope

The standard provenance object may contain:

```json
{
  "created_by": {
    "id": "01906c68-12b4-7dd1-b649-0f606babc501",
    "kind": "Party",
    "owning_context": "IdentityAccessGovernance"
  },
  "source_references": [],
  "transformations": [],
  "selection_rules": [],
  "material_omissions": [],
  "assumptions": [],
  "tool_versions": [],
  "limitations": []
}
```

A contract may require a subset or additional fields. Empty arrays shall not be interpreted as a claim of completeness unless the contract and applicable policy explicitly permit that claim.

### 12. Extensions are namespaced and losslessly preserved

Context-specific extension keys use reverse-domain or project-governed namespaces.

Example:

```json
{
  "extensions": {
    "okw.work-planning": {},
    "com.example.analytics": {}
  }
}
```

An extension shall not silently redefine a core field.

A consumer that does not understand an extension shall:

1. preserve it during a lossless round trip when authorized;
2. avoid claiming semantic validation of that extension;
3. reject the envelope only when the extension is marked required for the requested operation.

### 13. Required extensions are declared

An envelope may include:

```json
{
  "required_extensions": [
    "okw.work-planning"
  ]
}
```

A consumer shall reject an operation when it cannot interpret a required extension needed for that operation.

Unknown optional extensions remain opaque and preserved.

### 14. Markdown is a supported authoring representation

Markdown may remain the primary human authoring representation for prose documents.

A Markdown adapter shall produce or consume canonical semantic envelopes without losing:

- Document and Revision identity;
- Content Region identity;
- block ordering;
- managed references;
- annotations required by the contract;
- provenance declarations;
- authority and epistemic metadata;
- extension data required for round-trip editing.

Metadata may be represented through documented front matter, embedded semantic markers, sidecar files, or repository manifests.

The adapter format shall not make repository path the Resource identity.

### 15. Rendered outputs are derivative representations

HTML, PDF, DOCX, images, and other rendered outputs shall identify the exact source envelope or Publication Manifest from which they were produced.

A rendered output is not expected to round-trip into the complete authoring model unless its contract explicitly guarantees that behavior.

### 16. Content Regions use identity plus occurrence structure

A Content Region definition contains stable Region Identity and region-level semantics.

A Region Occurrence identifies where the Region appears in an exact Document Revision and includes anchoring information defined by ADR-HT-003.

Canonical serialization shall not equate a byte offset, line number, or editor node ID with stable Region Identity.

### 17. Relationships use independent envelopes

A consequential Relationship is serialized as its own `okw.relationship` envelope.

Minimum payload fields include:

- Relationship Type;
- source reference;
- target reference;
- originating context;
- intended meaning;
- applicable Revision selection;
- authority or policy basis where applicable;
- lifecycle representation defined by ADR-HT-004;
- authorization-aware discovery metadata defined by ADR-HT-005.

Embedding a Relationship summary inside another object does not replace the authoritative Relationship envelope.

### 18. Trails use ordered Step envelopes within a Trail Revision

A Trail Revision payload contains:

- Trail Identity;
- audience;
- purpose;
- ordered Trail Steps;
- selection modes;
- rationale;
- alternatives;
- known omissions;
- governing policies.

Each Trail Step contains stable Step Identity and a typed target reference.

Trail Step branching representation remains governed by ADR-HT-006.

### 19. Immutable Records are append-only semantic envelopes

An Immutable Record envelope shall include:

- Record Identity;
- record type;
- subject references;
- actor or originating agent;
- authority basis where applicable;
- effective time where applicable;
- recorded time;
- record-specific payload;
- provenance;
- correction, reversal, or supersession references where applicable.

A correction creates a new record. Serialization shall not overwrite the original record.

### 20. Projection Instances record exact reproducibility inputs

A Projection Instance envelope shall identify:

- Projection Instance Identity;
- exact Projection Definition Revision;
- source identities and exact Revisions or explicit temporal boundary;
- parameter values;
- applicable policy versions;
- tool version;
- generation time;
- output digest;
- completeness and limitation information.

### 21. Integrity metadata is explicit

An immutable envelope may include:

```json
{
  "integrity": {
    "canonicalization": "RFC8785",
    "digest_algorithm": "sha-256",
    "digest": "base64url-value"
  }
}
```

The digest is calculated over the canonicalized envelope with the `integrity.digest` value omitted or set according to the contract's digest procedure.

A contract shall define the exact digest input rule to avoid self-reference ambiguity.

### 22. Schema documents are informative validation artifacts

JSON Schema files may validate structural conformance and provide tooling support.

They do not fully define domain semantics, authority, lifecycle, or cross-context invariants. Those remain governed by the domain contracts, ADRs, and constitutional rules.

A schema identifier and schema version shall be recorded where used.

### 23. Lossless round trip is required for semantic editors

A conforming semantic editor or adapter shall pass this test:

```text
canonical envelope
→ parse
→ internal representation
→ serialize
→ canonicalize
```

The final canonical bytes shall equal the initial canonical bytes for all fields the editor is authorized to preserve.

An adapter may intentionally transform an envelope only by creating a new Revision or Derivative Representation with recorded provenance.

### 24. Independent reconstruction is required

A documented export containing canonical envelopes and referenced schemas shall be reconstructable by an implementation that does not share the producer's database or codebase.

The reconstruction shall preserve:

- principal identities;
- Revision relationships;
- Relationship endpoints and types;
- ordered structures;
- required extensions;
- provenance;
- authority references;
- integrity metadata.

### 25. Security and privacy filtering occurs before export

Canonical serialization does not grant disclosure authority.

Before export or cross-context delivery, the producing context shall apply authorization, consent, retention, governed-history, and inference-risk policy.

Filtered output shall not falsely claim completeness.

## Consequences

### Positive

- Semantics survive editor and implementation replacement.
- Canonical bytes support stable hashing, signing, and reproducibility.
- Bounded contexts retain internal implementation freedom.
- Unknown optional extensions can survive round trips.
- Markdown remains usable without becoming the only semantic authority.
- Exports can be independently reconstructed.

### Costs

- Every adapter must map to and from canonical envelopes.
- Exact decimal and temporal rules require disciplined contract design.
- Unknown-field preservation increases editor complexity.
- Canonicalization and digest procedures require conformance tests.
- Human-readable files and canonical bytes may use different property ordering and whitespace.

### Risks

- Teams may treat JSON Schema as the entire domain model.
- Implementations may discard unknown extensions.
- Editors may preserve visible text while losing semantic metadata.
- Derived display copies may be mistaken for authoritative source.
- Canonicalization differences may create false integrity failures.

These risks shall be controlled through shared fixtures, round-trip tests, contract documentation, and adapter certification.

## Rejected Alternatives

### Markdown as the universal canonical format

Rejected because Markdown does not natively and consistently preserve all required identity, Relationship, provenance, extension, and typed-record semantics across tools.

### Database schema as semantic authority

Rejected because it couples the domain to one implementation and prevents independent reconstruction.

### JSON-LD as a mandatory universal representation

Rejected as the mandatory base because contexts need deterministic contract-owned payloads and not every consumer requires RDF processing.

JSON-LD-compatible context mappings may be provided as optional interoperability projections.

### YAML as the canonical integrity format

Rejected because parser behavior, scalar typing, aliases, and serialization variation create avoidable canonicalization and interoperability risk.

### Binary-only serialization

Rejected as the sole canonical representation because the project requires documented, inspectable, implementation-independent exchange.

Binary projections may be used for performance when mapped losslessly to canonical envelopes.

## Verification Expectations

Implementation verification shall eventually demonstrate:

1. deterministic canonicalization of shared fixtures;
2. digest equality across independent implementations;
3. lossless parse-and-serialize round trips;
4. preservation of unknown optional extensions;
5. rejection of unsupported required extensions;
6. exact decimal preservation;
7. ordered Trail Step preservation;
8. Markdown-to-envelope-to-Markdown semantic preservation;
9. independent export reconstruction;
10. safe filtering without false completeness claims.

# Architecture Decision: Content Region Anchoring

**Identifier:** ADR-HT-003

**Status:** Accepted

**Date:** 2026-06-27

**Project:** Organizational Knowledge and Work System

## Context

A Content Region is a stably identified and explicitly bounded portion of a Source Document. It may be referenced, reused, reviewed, validated, discussed, or included independently of the whole Document.

Region Identity must survive ordinary insertion, deletion, formatting changes, renaming, and movement that do not conceptually replace the Region.

At the same time, the system must never silently retarget a Region to similar but incorrect content after:

- surrounding edits;
- movement;
- copied text;
- repeated text;
- block splitting or merging;
- concurrent revisions;
- conflict resolution;
- deletion;
- conversion between textual and visual editors;
- repository or storage migration.

ADR-HT-001 establishes stable Region Identity and revision-specific Region Occurrences. ADR-HT-002 establishes versioned JSON semantic envelopes and requires Content Region identity plus occurrence structure.

This decision defines how a Region is anchored and resolved without making line numbers, byte offsets, repository paths, or text similarity the durable identity.

## Decision

### 1. Region identity, version, and occurrence are distinct

The system shall represent three related concepts:

1. **Content Region** — the continuing conceptual identity.
2. **Region Version** — an immutable semantic state of that Region.
3. **Region Occurrence** — the location of one Region Version within one exact Document Revision.

A Region Occurrence shall identify:

- Occurrence Identity;
- Region Identity;
- Region Version Identity;
- Document Identity;
- exact Document Revision Identity;
- start boundary;
- end boundary;
- anchor evidence;
- resolution status;
- recorded time;
- provenance.

A new Document Revision produces new Region Occurrences for Regions present in that Revision, even when a Region Version is unchanged.

### 2. The canonical document model uses stable semantic block identity

Every addressable semantic block in a canonical Document Revision shall have stable Block Identity.

Examples of semantic blocks include:

- heading;
- paragraph;
- list item;
- table;
- table row or cell where independently addressable;
- definition;
- quotation;
- code block;
- figure caption;
- structured visual node represented in the canonical model.

Block Identity survives ordinary edits and movement when the block remains the same conceptual block.

A copied block receives new Block Identity unless an explicit move operation transfers the original block rather than copying it.

Editor node identifiers, DOM identifiers, repository line numbers, and database row keys do not replace Block Identity.

### 3. Region boundaries use a half-open semantic range

A Region Occurrence uses a half-open range:

```text
[start boundary, end boundary)
```

The start boundary is inclusive. The end boundary is exclusive.

Each boundary identifies:

- Block Identity;
- an intra-block offset when the boundary does not coincide with the block edge;
- offset unit;
- optional semantic child path for structured blocks.

The default text offset unit is a Unicode code-point index into the exact canonical block text.

Byte offsets, UTF-16 code-unit offsets, rendered pixel positions, and visual coordinates may be recorded as adapter metadata but are not the canonical text offset.

### 4. Full-block Regions are preferred

Where practical, Content Regions should align to whole semantic blocks.

Whole-block Regions are more resilient, explainable, and portable than arbitrary character spans.

Sub-block Regions are permitted when the use case requires exact phrases, clauses, cells, or inline structures. They shall include stronger quote and context evidence because intra-block edits create greater ambiguity.

### 5. Every occurrence carries a selector bundle

A Region Occurrence shall carry multiple selectors and evidence types rather than depend on one locator.

Minimum selector bundle:

```json
{
  "start": {
    "block_id": "01906d21-72fb-73c0-8df8-3cf67f6bd294",
    "offset": 0,
    "offset_unit": "unicode-code-point"
  },
  "end": {
    "block_id": "01906d21-72fb-73c0-8df8-3cf67f6bd294",
    "offset": 86,
    "offset_unit": "unicode-code-point"
  },
  "quote": {
    "exact": "Participants described inconsistent approval guidance across teams.",
    "prefix": "Interview synthesis: ",
    "suffix": " The issue affected onboarding."
  },
  "structure": {
    "ancestor_block_ids": [],
    "document_order_hint": 12
  },
  "integrity": {
    "canonical_text_sha256": "base64url-or-hex-value"
  }
}
```

The primary selector is the stable Block Identity boundary pair.

Quote, context, structural path, order, and digest fields are corroborating or recovery evidence. They are not independent authority to retarget a Region silently.

### 6. Exact historical resolution is deterministic

A Region Occurrence targeting an exact Document Revision resolves only against that Revision.

If the referenced Revision and blocks remain available, resolution uses the recorded Block Identities and offsets directly.

A later Revision does not change the meaning or location of a historical occurrence.

Historical occurrences remain immutable.

### 7. Carry-forward to a new Document Revision uses explicit edit lineage

When a new Document Revision is created, the authoring adapter or revision service shall produce block-lineage information describing, where applicable:

- block preserved;
- block moved;
- block edited;
- block split;
- blocks merged;
- block copied;
- block deleted;
- new block created;
- lineage unresolved.

Region carry-forward uses this lineage before any content matching.

If both boundaries map unambiguously through preserved or moved Block Identities, the Region carries forward automatically.

### 8. Resolution uses a strict ordered strategy

A Region resolver shall evaluate candidates in this order:

1. exact Document Revision and exact Block Identities;
2. explicit block-lineage mapping into a descendant Revision;
3. exact canonical text digest within the permitted lineage scope;
4. exact quote with prefix and suffix context;
5. exact quote without full context, only when unique within the permitted scope;
6. human review or explicit repair.

Fuzzy text similarity may rank repair candidates for human review. It shall not independently authorize automatic retargeting of a consequential Region.

### 9. Resolution fails closed on ambiguity

Automatic resolution is permitted only when one candidate satisfies the applicable rules.

If multiple candidates are plausible, the status is `Ambiguous`.

If no candidate exists, the status is `Orphaned` or `Deleted`, according to known edit lineage.

The system shall not choose the first candidate, nearest candidate, most popular candidate, or highest fuzzy score without explicit policy and review.

### 10. Resolution status is explicit

Supported minimum statuses are:

- `Resolved` — boundaries resolve directly in the expected location;
- `Relocated` — identity resolves unambiguously after movement;
- `Recovered` — stable block lineage was unavailable, but exact corroborating evidence produced one permitted candidate;
- `Modified` — Region Identity continues with a new Region Version;
- `Ambiguous` — more than one plausible candidate exists;
- `Orphaned` — no current occurrence can be established;
- `Deleted` — explicit lineage records deletion;
- `Split` — the original Region was explicitly split into new Regions;
- `Merged` — Regions were explicitly combined into a new Region;
- `Conflict` — concurrent revisions require resolution;
- `Restricted` — the occurrence exists but cannot be disclosed to the requester.

A Projection may summarize these statuses but shall preserve the underlying evidence and policy basis.

### 11. Ordinary surrounding edits preserve Region Identity

Insertion, deletion, or reordering outside a Region does not create a new Region Version when the Region's canonical semantic content and meaning are unchanged.

A new Document Revision still creates a new Region Occurrence.

Changes to line number, byte position, visual placement, heading numbering, or neighboring prose do not by themselves change Region Identity.

### 12. Formatting-only changes may preserve Region Version

A formatting change that leaves canonical semantic structure and text unchanged may preserve Region Version.

Examples may include:

- renderer-specific whitespace;
- typography;
- visual color;
- non-semantic layout;
- editor metadata that is not part of the contract.

A format change that alters semantic structure, accessibility meaning, code behavior, table interpretation, or another contract-owned property creates a new Region Version.

### 13. Content edits may preserve Region Identity but create a new Region Version

When content changes but remains the same conceptual Region, Region Identity is preserved and a new Region Version is created.

The new occurrence shall record:

- prior Region Version;
- exact Document Revision;
- changed boundaries when applicable;
- change description;
- author or Automated Agent;
- provenance;
- applicable review or approval.

Conceptual continuity is an authored or governed decision, not an inference from text similarity alone.

### 14. Movement preserves identity through explicit transfer

Moving a Region within a Document preserves Region Identity and Region Version when semantic content is unchanged.

Moving a Region to another Document may preserve Region Identity only through an explicit transfer operation that records:

- source Document and Revision;
- destination Document and Revision;
- source and destination occurrences;
- transfer authority;
- whether the source occurrence remains, is withdrawn, or is superseded.

Copying content into another Document creates new Block and Region Identities linked through `CopiedFrom` or `DerivedFrom` Relationships.

### 15. Split creates new Region identities

When one Region becomes two or more independently meaningful Regions:

- the original Region retains its historical identity;
- new Region Identities are created for the resulting Regions;
- the original Region is marked `Split` or superseded in current projections;
- explicit `SplitInto` Relationships identify the new Regions;
- each new Region records source-span provenance.

A resolver shall not silently assign the original Region Identity to one child merely because that child contains most of the original text.

### 16. Merge creates a new Region identity

When two or more Regions become one independently meaningful Region:

- a new Region Identity is created;
- source Regions retain historical identity;
- `MergedFrom` Relationships identify all source Regions;
- the new Region records selection and transformation provenance.

A merge shall not reuse one source Region Identity while silently absorbing the others.

### 17. Nested Regions are permitted; crossing overlaps require explicit contract support

Regions may be:

- disjoint;
- equal in boundary only when their semantic roles are explicitly distinct;
- nested.

Crossing overlaps are prohibited by default:

```text
A starts → B starts → A ends → B ends
```

A bounded-context contract may permit crossing overlap only when it defines deterministic editing, rendering, and lifecycle behavior.

### 18. Concurrent revisions preserve branch-local occurrences

When Document history branches, the same Region Identity may have branch-local Region Versions and Occurrences.

A merge shall evaluate Region lineage explicitly.

If both branches changed the same Region incompatibly, the merged state is `Conflict` until an authorized reconciliation creates:

- a selected Region Version;
- a new reconciled Region Version; or
- explicit split, merge, withdrawal, or replacement records.

The system shall not choose a branch solely by timestamp or commit order.

### 19. Deletion preserves historical addressability subject to policy

When explicit lineage records that a Region was removed, its current status is `Deleted`.

Historical Region Occurrences remain addressable subject to retention, authorization, consent, and governed-history policy.

If policy removes historical content or identifying residue, the Region may become unresolvable. Its identifier is never reassigned.

### 20. Restoration requires explicit continuity

A later reappearance of identical or similar text does not automatically restore a deleted Region.

Reusing the original Region Identity requires an explicit restoration decision stating that the same conceptual Region has returned.

Otherwise, a new Region Identity is created with an optional `DerivedFrom` or `Resembles` Relationship.

### 21. Generated and visual documents use semantic nodes

For generated or visual Documents, a Content Region anchors to canonical semantic nodes rather than rendered coordinates.

Examples include:

- diagram node identity;
- chart series or annotation identity;
- table cell identity;
- structured canvas object identity;
- accessible text node identity.

Rendered bounding boxes and pixel coordinates may support interaction but do not establish durable Region identity.

### 22. Markdown adapters preserve block and Region markers

A Markdown adapter may use:

- embedded semantic comments;
- front matter;
- sidecar manifests;
- repository-level semantic indexes;
- a combination of these mechanisms.

The canonical envelope remains authoritative.

A Markdown round trip shall preserve Block, Region, Region Version, and managed-reference identities required by the contract.

Removal or corruption of a marker shall produce a repair state rather than silent identity reassignment.

### 23. Anchor repair is a consequential operation

Manual anchor repair shall record:

- affected Region Identity;
- prior occurrence;
- selected new occurrence;
- repair reason;
- candidate evidence;
- repairing Party or Automated Agent;
- authority;
- time;
- whether a new Region Version was created.

Automated tools may propose repairs. Consequential repairs require the review policy defined by the owning context.

### 24. Authorization applies to resolution and diagnostics

A resolver shall not disclose restricted content, candidate counts, quote text, structural paths, or existence through diagnostics beyond policy.

An unauthorized requester may receive a policy-appropriate `Restricted`, `Unavailable`, or generalized result.

The absence of a visible candidate shall not be represented as proof that no restricted candidate exists.

## Resolution Algorithm

A conforming resolver follows this decision procedure:

```text
1. Validate the occurrence and requested Document Revision.
2. Enforce authorization before exposing content or diagnostics.
3. Resolve exact Block Identities and offsets when available.
4. Otherwise follow explicit block-lineage mappings.
5. Otherwise search only within the permitted scope using exact digest and quote evidence.
6. If exactly one candidate satisfies required evidence, return Recovered.
7. If more than one candidate satisfies required evidence, return Ambiguous.
8. If deletion lineage exists, return Deleted.
9. If no candidate exists, return Orphaned.
10. Never silently retarget through fuzzy similarity alone.
```

## Consequences

### Positive

- Regions survive ordinary editing and movement without relying on line numbers.
- Exact historical references remain deterministic.
- Duplicate text does not cause silent mis-targeting.
- Split, merge, copy, deletion, and restoration preserve accountable identity history.
- Textual and visual editors can share one semantic anchoring model.
- Recovery evidence supports damaged-marker repair without becoming hidden authority.

### Costs

- Editors and importers must preserve Block Identity and edit lineage.
- Every Document Revision containing Regions requires occurrence records.
- Sub-block Regions require careful Unicode offset handling.
- Merge tooling must understand Region lineage.
- Manual repair workflows and authorization-sensitive diagnostics are required.

### Risks

- Block Identity may be preserved across conceptual replacement.
- Editors may accidentally copy hidden identifiers.
- Quote recovery may produce false candidates in repetitive documents.
- Unicode offset implementations may disagree.
- Sidecar and source markers may diverge.

These risks are controlled by explicit edit operations, canonical envelopes, collision checks, fail-closed ambiguity handling, and conformance fixtures.

## Rejected Alternatives

### Line and column numbers as durable anchors

Rejected because surrounding edits and formatting changes invalidate them.

### Byte offsets as durable anchors

Rejected because encoding and preceding-content changes invalidate them.

### Text quote alone

Rejected because identical text may occur more than once and text changes break the anchor.

### Fuzzy similarity with automatic best match

Rejected because a plausible match can silently target the wrong content.

### Editor-specific node identity as canonical identity

Rejected because it couples durable references to one editor implementation.

### Content hash as Region Identity

Rejected because legitimate Region edits must preserve conceptual identity while changing content.

## Verification Expectations

Implementation verification shall eventually demonstrate:

1. insertion before a Region preserves identity;
2. movement preserves identity and reports relocation;
3. formatting-only edits preserve Region Version when semantics are unchanged;
4. content edits create a new Region Version without replacing Region Identity;
5. duplicate quote candidates produce `Ambiguous`;
6. deletion produces `Deleted` or `Orphaned` without retargeting;
7. split and merge create new Region Identities and lineage Relationships;
8. concurrent conflicting edits produce `Conflict`;
9. copied markers do not duplicate Region Identity silently;
10. Markdown and visual adapters preserve semantic anchors;
11. unauthorized diagnostics do not leak restricted candidate information;
12. independent implementations resolve shared fixtures consistently.

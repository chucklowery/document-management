# Change Processing and Consistency EARS Requirements

**Project:** Organizational Knowledge and Work System

## REQ-SM-001 — Canonical Change Event

**EARS:** When a durable change may affect retrieval, Relationships, Projections, subscriptions, or delivered outputs, the system shall create or derive a Change Event containing stable event identity, affected identities, applicable Revisions, source, authority, temporal information, causation, idempotency, and ordering scope.

**Stories:** US-065, US-068

## REQ-SM-002 — Idempotent Processing

**EARS:** When the same Change Event or idempotency key is delivered more than once within its declared scope, the system shall not create duplicate domain effects and shall record the duplicate handling outcome.

**Stories:** US-065, US-067

## REQ-SM-003 — Explicit Ordering Scope

**EARS:** When processing order affects meaning, the producing contract shall identify the Ordering Scope and sequence rule, and the system shall not imply ordering outside that scope.

**Stories:** US-065

## REQ-SM-004 — Late-Arriving Information

**EARS:** When information arrives after later-recorded information, the system shall preserve its known effective, observation, or occurrence time and shall recalculate affected current Projections without rewriting prior records.

**Stories:** US-065

## REQ-SM-005 — Invalid Input Quarantine

**EARS:** If incoming information fails contract, integrity, authorization, or semantic validation and cannot be safely rejected at the source boundary, then the system shall create a Quarantine Record and shall not expose the input as accepted managed information.

**Stories:** US-066

## REQ-SM-006 — Processing Attempt Record

**EARS:** When consequential event processing succeeds, retries, fails, is quarantined, or reaches terminal disposition, the system shall record the event boundary, processing definition version, attempt time, outcome, produced effects, diagnostics, and retry eligibility.

**Stories:** US-066, US-067, US-068

## REQ-SM-007 — Bounded Retry and Dead-Letter Disposition

**EARS:** When a processing failure is retryable, the system shall apply a bounded retry policy; when retries are exhausted or unsafe, the system shall create a terminal failure or dead-letter disposition with responsible stewardship and escalation.

**Stories:** US-066, US-068

## REQ-SM-008 — Observable Propagation

**EARS:** When an accepted change requires downstream effects, the system shall expose authorized processing checkpoints, Consistency State, measured or estimated Propagation Lag, failures, and affected derived representations.

**Stories:** US-068

## REQ-SM-009 — Authorized Replay

**EARS:** When an authorized operator replays Change Events, the system shall record the event boundary, processing definition, reason, actor, start and completion checkpoints, outcome, and duplicate-prevention evidence.

**Stories:** US-067

## REQ-SM-010 — Authorized Backfill

**EARS:** When an authorized operator backfills historical source information, the system shall identify the source boundary, contract and mapping versions, temporal interpretation, idempotency scope, generated changes, and completion status.

**Stories:** US-067

## REQ-SM-011 — Read-Your-Writes for Acknowledged Change

**EARS:** Where a workflow declares read-your-writes consistency, when the system acknowledges a change, subsequent authorized reads in that workflow shall resolve the acknowledged Revision or explicitly disclose that propagation is pending.

**Stories:** US-059, US-068

## REQ-SM-012 — Expected Base Revision

**EARS:** When a Party or Automated Agent submits a change to an existing Resource, the system shall require or derive the Expected Base Revision and shall compare it with the current applicable Revision before acceptance.

**Stories:** US-069

## REQ-SM-013 — Preserve Concurrent Contributions

**EARS:** If concurrent acknowledged contributions are based on incompatible Revisions, then the system shall preserve each contribution and shall not silently overwrite either contribution.

**Stories:** US-069

## REQ-SM-014 — Reconciliation Record

**EARS:** When concurrent contributions are merged, retained as alternatives, rejected, superseded, or deferred, the system shall create a Reconciliation Record identifying the contributions, base Revisions, authority, policy, rationale, and resulting Revisions.

**Stories:** US-069

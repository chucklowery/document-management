# Interactive Control Authoring EARS Requirements

**Project:** Organizational Knowledge and Work System

## REQ-IC-001 — Visible Active Configuration and Mode

**EARS:** When an authorized user opens a Control Surface, the system shall identify the active Configuration Revision, affected scope, connected inputs and outputs, operating mode, material pending work, freshness, and actions available to that user.

**Stories:** US-102

## REQ-IC-002 — Draft Before Consequential Activation

**EARS:** When a user begins a consequential configuration change, the system shall create or resume a Configuration Draft and shall not alter active behavior until an identified Configuration Revision is authorized and activated.

**Stories:** US-103, US-111

## REQ-IC-003 — Contract-Aware Rule and Mapping Authoring

**EARS:** When a user authors a Rule Definition or Mapping Definition, the system shall present fields, types, required values, defaults, conversions, omissions, temporal semantics, and permitted effects from identified Input Contract and Output Contract Revisions.

**Stories:** US-103

## REQ-IC-004 — Incremental Validation and Error Localization

**EARS:** When a draft value, rule, mapping, example, or dependency can be validated, the system shall validate it as early as practical and shall associate each finding with the exact affected element, severity, governing contract or policy, and authorized repair guidance.

**Stories:** US-103, US-104

## REQ-IC-005 — Preserve Valid Draft Work

**EARS:** If one part of a Configuration Draft is invalid or unavailable, then the system shall preserve unrelated valid draft work and shall not require a full reset unless the retained state cannot be interpreted reliably.

**Stories:** US-104, US-111

## REQ-IC-006 — Representative Example Evaluation

**EARS:** When configurable behavior is tested, the system shall evaluate it against an identified Example Set containing applicable representative, boundary, missing-value, invalid, restricted, and exceptional cases and shall itemize outcomes and unevaluated cases.

**Stories:** US-103, US-105

## REQ-IC-007 — Impact Preview Before Consequential Change

**EARS:** When a consequential Configuration Revision is proposed, the system shall provide an authorized Impact Preview identifying evaluated scope, affected identities, channels, users, dependencies, predicted effects, warnings, failures, unknowns, generated time, and limitations, or shall explain why a preview is unavailable.

**Stories:** US-105

## REQ-IC-008 — Simulation Does Not Commit Live Effects

**EARS:** While a Simulation Run or dry run is executing, the system shall avoid live-state change and external delivery unless the run explicitly uses an isolated destination identified as non-production.

**Stories:** US-105

## REQ-IC-009 — Stable Scope from Preview to Execution

**EARS:** When an Activation Plan is approved, the system shall preserve the Stable Selection Set, Configuration Revision, contract Revisions, authorization context, and temporal boundary used for approval or shall re-evaluate them according to an explicit selection and drift policy before execution.

**Stories:** US-105, US-106

## REQ-IC-010 — Execution Drift Decision

**EARS:** If material Execution Drift is detected after preview or approval, then the system shall stop, reduce scope, require renewed preview, request renewed confirmation or approval, or continue only under an identified policy that records the difference and authority.

**Stories:** US-105, US-106, US-107

## REQ-IC-011 — Prompt Acknowledgement with Provisional Status

**EARS:** When a user requests an authorized operation, the system shall promptly acknowledge receipt with request identity, accepted scope, current Execution State, and the distinction between accepted intent and completed durable effects.

**Stories:** US-107

## REQ-IC-012 — Durable Progress and Terminal Accounting

**EARS:** While an operation is pending or running, the system shall expose authorized progress, checkpoints, measured or estimated remaining work, delay, partial outcomes, and failures; when it terminates, the system shall account for every selected item or declared aggregate boundary.

**Stories:** US-107

# Information Retrieval and System Management Domain Model

**Project:** Organizational Knowledge and Work System

## 1. Purpose

This document defines the model required to retrieve changing information, process inputs, propagate changes, deliver outputs, reconcile concurrent contributions, and operate the system recoverably.

It extends the Shared Foundation without making indexes, queues, caches, or delivery systems authoritative sources.

## 2. Retrieval Model

### 2.1 Corpus Definition

A Corpus Definition is a versioned Resource identifying eligible contexts, Resource and record kinds, fields, extraction rules, temporal policy, classification constraints, inclusion and exclusion rules, and completeness assumptions.

**Invariants:**

1. Corpus eligibility is explicit.
2. A Corpus Definition does not grant authorization.
3. Material policy change creates a new Revision.
4. Historical retrieval records the applicable Revision.

### 2.2 Ranking Policy

A Ranking Policy is a versioned Resource identifying permitted relevance factors, tie-breaking and grouping, freshness treatment, authority and epistemic treatment, relationship expansion, diversification, prohibited signals, and evaluation baseline.

**Invariants:**

1. Popularity, usage, graph centrality, and feedback do not establish truth or authority.
2. Material ordering-policy change creates a new Revision.
3. A Ranking Policy cannot bypass authorization.
4. One Ranking Policy is not the sole valid view.

### 2.3 Index Revision

An Index Revision is an immutable derived record identifying its Corpus Definition Revision, extraction definition, source boundary, authorization policy, generation times, tool version, integrity, completeness, failures, and exclusions.

**Invariants:**

1. An Index Revision is derived.
2. It is rebuildable from permitted durable sources and definitions.
3. It does not become authoritative.
4. Its completeness and freshness are explainable.

### 2.4 Query Session and Retrieval Projection

A Query Session may contain Queries, filters, refinements, views, selections, and feedback.

A Retrieval Projection records or can reproduce its Query or protected digest, task context, Corpus Definition Revision, Index Revision or source boundary, Ranking Policy Revision, authorization and policy versions, temporal boundary, generated time, returned Searchable Units, explanations, and limitations.

**Invariants:**

1. Query content follows sensitivity policy.
2. Results refer to managed identities and applicable Revisions or occurrences.
3. Current and historical retrieval are distinguishable.
4. Restricted candidates do not leak through order, counts, gaps, or explanations.

### 2.5 Relevance Judgment and Evaluation Set

A Relevance Judgment associates a Query or class, task, assessor, Searchable Unit, judgment, time, rationale where required, and applicable corpus and policy versions.

An Evaluation Set is a governed Resource containing representative tasks and judgments.

**Invariants:**

1. Relevance is contextual.
2. A judgment does not transfer authority.
3. Evaluation changes are versioned.
4. Ranking adoption records the evaluation evidence used.

## 3. Input and Propagation Model

### 3.1 Input Contract

An Input Contract is a versioned Resource defining source authority, authentication, supported contracts, validation, mapping, idempotency, ordering, temporal interpretation, retry, quarantine, deletion, correction, checkpoints, and backfill.

### 3.2 Change Event

A Change Event contains Event Identity, event type, source, affected identities, applicable Revisions, temporal values, causation and correlation, idempotency key, Ordering Scope, payload or reference, integrity, authority, and policy context.

**Invariants:**

1. Repeated delivery does not create duplicate domain effects.
2. Order is guaranteed only within the declared Ordering Scope.
3. A Change Event is not authoritative state.
4. A durable event is not silently discarded.

### 3.3 Processing Attempt and Quarantine

A consequential processing attempt records its event or source boundary, processing definition, time, outcome, produced effects, retry eligibility, diagnostics, and quarantine or terminal disposition.

**Invariants:**

1. Failures remain observable.
2. Retry does not duplicate acknowledged effects.
3. Quarantined information is not searchable or delivered as accepted information.
4. Correction preserves original failure history.

### 3.4 Consistency Projection

A Consistency Projection explains the relationship among authoritative source state, event checkpoints, backlink indexes, retrieval indexes, materialized Projections, caches, subscriptions, and deliveries. It records state, lag, failures, rebuild status, and known gaps.

## 4. Output and Delivery Model

### 4.1 Output Manifest

An Output Manifest identifies output identity and type, exact source Revisions or temporal boundary, Projection or generation definition, parameter and policy versions, authorization scope, generated time, completeness, limitations, integrity, and supersession, withdrawal, or expiration state.

### 4.2 Subscription

A Subscription is a versioned Resource containing recipient, triggering Saved Search, Query, event, or schedule, Output Contract, channel, frequency, grouping, attention policy, effective period, acknowledgement, and escalation.

### 4.3 Delivery Attempt

A Delivery Attempt is an immutable record containing Output Manifest, Subscription or request, destination, channel, idempotency key, attempt number and time, outcome, acknowledgement, diagnostics, and retry or terminal disposition.

**Invariants:**

1. Delivery does not establish acceptance or adoption.
2. Duplicate attempts do not create duplicate intended deliveries where idempotency applies.
3. Restricted or withdrawn output follows governance after generation.
4. Delivery history remains auditable subject to retention.

## 5. Concurrency Model

A proposed change contains an Expected Base Revision. If the current applicable Revision differs, the owning context may reject and require rebase, apply a documented deterministic merge, create a multi-parent Revision, retain alternatives, or require authorized reconciliation.

A Reconciliation Record identifies affected contributions, policies, actors, rationale, and resulting Revisions.

**Invariants:**

1. Acknowledged work is not silently lost.
2. Automatic merge is limited to semantically safe fields and Relationship types.
3. Human and automated contributions retain separate provenance.
4. Reconciliation does not rewrite historical Revisions.
5. Reconciliation authority is explicit.

## 6. Operations Model

### 6.1 Operational Service Objective

An Operational Service Objective identifies capability, scope, measurement definition, target, window, exclusions, owner, breach policy, and review period.

### 6.2 Recovery Plan and Exercise

A Recovery Plan identifies backups, durable sources, event boundaries, reconstruction order, credentials and policy dependencies, RPO, RTO, and verification.

A Recovery Exercise records environment, inputs, times, restored boundaries, failures, data loss if any, and evidence.

### 6.3 Degraded Mode

A Degraded Mode declaration identifies affected capability, start time, reason, permitted operations, stale or unavailable representations, user disclosure, owner, and exit criteria.

**Invariants:**

1. Degraded operation does not weaken authorization.
2. Stale results are marked.
3. Writes are disabled when durability or authority cannot be preserved.
4. Exit is recorded and verified.

## 7. Context Ownership

- Information Access and Retrieval owns Query, Query Session, Corpus Definition, Ranking Policy, Evaluation Set, Retrieval Projection, and Saved Search semantics.
- Shared Graph Foundation owns durable addressing and relationship conventions.
- Owning domain contexts remain authoritative for source meaning and lifecycle.
- Integration and External Systems owns connector mappings and synchronization records.
- Identity, Access, and Governance owns authorization and disclosure policy.
- System Operations and Delivery owns processing observability, recovery coordination, delivery records, service objectives, and degraded-mode declarations.
- No context mutates another context's authoritative record through an index or delivery mechanism.

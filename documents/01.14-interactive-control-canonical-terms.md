# Interactive Control and End-User Configuration Canonical Terms

**Project:** Organizational Knowledge and Work System

## Purpose

These terms describe user-configurable behavior without prescribing a user-interface framework, rules engine, workflow product, or programming language.

### Control Surface

An authorized interaction view through which a user inspects system state, prepares a change, initiates an operation, or responds to an outcome.

### Active Configuration

The identified Configuration Revision currently governing a declared scope.

### Configuration Draft

A mutable working resource in which an authorized author prepares a proposed rule, mapping, filter, transformation, subscription, control policy, or related setting before activation.

### Configuration Revision

An immutable acknowledged version of configurable behavior, including owner, scope, contracts, parameters, dependencies, validation evidence, policy, and lifecycle state.

### Rule Definition

A versioned, executable condition-and-action definition with declared inputs, outputs, types, authority, evaluation semantics, and limits.

### Mapping Definition

A versioned transformation from an identified input contract to an identified target contract, including field correspondence, defaults, conversions, omissions, validation, and error behavior.

### Example Set

A versioned collection of representative, boundary, invalid, restricted, and exceptional examples used to explain, validate, or simulate configurable behavior.

### Impact Preview

A non-authoritative projection of the objects, channels, dependencies, users, policies, and expected outcomes that a proposed change may affect.

### Simulation Run

A recorded evaluation of a proposed Configuration Revision against an identified Example Set or bounded source snapshot without committing production effects.

### Activation Plan

A versioned plan identifying the Configuration Revision, target scope, approvals, start condition, rollout stages, verification checks, stop conditions, rollback or compensation path, and owner.

### Rollout Stage

A bounded activation step with an identified audience, channel, object set, percentage, time window, verification criteria, and continuation decision.

### Stable Selection Set

An identified set of target identities and applicable Revisions preserved from preview through execution, or revalidated according to an explicit drift policy.

### Execution Drift

A material difference between the scope, dependencies, source state, policy, or predicted effects used for preview and those present at execution time.

### Execution State

An explainable state of requested work, such as Draft, Validating, Ready, Awaiting Approval, Scheduled, Running, Paused, Cancelling, Completed, Partially Completed, Failed, Compensating, Reversed, or Unknown.

### Reversal Plan

A versioned definition for restoring a prior active configuration or otherwise reversing reversible effects while preserving history.

### Compensation Action

A new accountable action that mitigates or counteracts an effect that cannot be erased or literally undone.

### Manual Override

An authorized, time-bounded departure from ordinary automated behavior with declared scope, reason, approver, safeguards, expiration, and review requirement.

### Confirmation Policy

A policy that determines when explicit confirmation, reauthentication, peer approval, delay, or typed acknowledgement is required based on consequence rather than interface habit.

### Work Session

A recoverable record of an author's unsaved or draft interaction state, including scope, selections, field values, validation findings, and last durable checkpoint.

## Usage Rules

1. An Impact Preview is not evidence that execution completed.
2. A Simulation Run does not grant production authorization.
3. A restored configuration creates a new acknowledged state and does not delete the replaced Revision.
4. A Manual Override expires or is explicitly renewed; it is not an undocumented permanent setting.
5. Simplified controls do not change contract ownership or source authority.
6. An Execution State of Completed applies only to the declared boundary and required effects.
